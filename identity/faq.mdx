Quick answers to common questions plus troubleshooting for integration issues.

## Common Issues & Solutions

<AccordionGroup>
<Accordion title="ðŸ”„ Verification session expired (24 hours)">


<Accordion title="âŒ Verification failed - poor document quality">
  **Problem**: Most common verification failure
  
  **Solutions**:
  - Better lighting (avoid shadows and glare)
  - Ensure entire document is in frame
  - Use flat, dark background surface
  - Make sure all text is clearly readable
  
  **User messaging**: "Try retaking photos with better lighting and ensure all text is visible"
</Accordion>

<Accordion title="âš ï¸ Network errors during status polling">
  **Problem**: API timeouts or network failures
  
  **Solution**: Implement retry logic with exponential backoff
  ```typescript
  const checkStatusWithRetry = async (walletAddress, retries = 3) => {
    try {
      return await sdk.checkIdentityStatus(walletAddress);
    } catch (error) {
      if (retries > 0) {
        await new Promise(resolve => setTimeout(resolve, 2000));
        return checkStatusWithRetry(walletAddress, retries - 1);
      }
      throw error;
    }
  };
  ```
</Accordion>

<Accordion title="ðŸ” 'No verification session found' error">
  **Problem**: Trying to check status for non-existent session
  
  **Solutions**:
  - Verify wallet address is correct
  - Check if session was created successfully
  - Handle case where user hasn't started verification yet
</Accordion>
</AccordionGroup>

## Development Questions

<AccordionGroup>
<Accordion title="How do I handle Self.xyz service reliability?">
  **Service uptime**: Self.xyz maintains 99.9% uptime with global CDN infrastructure.
  
  **Graceful degradation approach**:
  ```typescript
  const startVerificationWithFallback = async (walletAddress, disclosures) => {
    try {
      return await sdk.verifyIdentity({ walletAddress, disclosures });
    } catch (error) {
      if (error.message.includes('service unavailable')) {
        // Show user-friendly message and allow retry later
        return {
          error: 'service_unavailable',
          message: 'Identity verification is temporarily unavailable. Please try again in a few minutes.',
          retryAfter: 300000 // 5 minutes
        };
      }
      throw error; // Re-throw other errors
    }
  };
  ```
  
  **Best practices**:
  - Always provide fallback to standard registration
  - Show clear status messages during outages
  - Implement retry logic with exponential backoff
</Accordion>

<Accordion title="How do I handle 24-hour session expiration UX?">
  **User experience approach**:
  
  ```typescript
  const handleExpiredSession = async (walletAddress) => {
    // Check if session is close to expiring (within 2 hours)
    const status = await sdk.checkIdentityStatus(walletAddress);
    const expiresAt = new Date(status.details.expiresAt);
    const now = new Date();
    const hoursUntilExpiry = (expiresAt - now) / (1000 * 60 * 60);
    
    if (hoursUntilExpiry < 2) {
      // Warn user and suggest completing soon
      showExpirationWarning(hoursUntilExpiry);
    }
    
    if (status.details.status === 'expired') {
      // Auto-start new session
      return sdk.verifyIdentity({ walletAddress, disclosures });
    }
  };
  ```
  
  **UX recommendations**:
  - Warn users 2 hours before expiration
  - Auto-restart expired sessions when user returns
  - Don't make users manually handle expiration
</Accordion>

<Accordion title="What edge cases should I plan for?">

  **Network interruption during verification**:
  ```typescript
  // Implement robust status checking

    for (let i = 0; i < maxRetries; i++) {
      try {
        return await sdk.checkIdentityStatus(walletAddress);

    }
  ```

  **User closes app mid-verification**:
  - Session persists for 24 hours
  - User can return and complete later
  - Poll status when user returns to app
  
  **Multiple devices/tabs**:
  - Verification sessions are wallet-specific
  - Status updates across all tabs polling same wallet
  - Handle concurrent polling gracefully
  
  **Document rejection scenarios**:

</Accordion>

  - **Zero-knowledge proofs**: The app generates cryptographic proofs without exposing raw document data
  - **User control**: Users choose exactly what information to share for each verification
  - **One-time setup**: Once installed, users can verify identity for any Self.xyz-integrated service
  
  **Download links**:
  - [iOS App Store](https://apps.apple.com/app/self-crypto-super-app/id1459811885)

<Accordion title="What is Self.xyz and why is it required?">
  **Self.xyz** is a zero-knowledge identity verification platform that allows users to prove their identity without exposing sensitive personal information. Trust Engine integrates with Self.xyz to provide privacy-preserving identity verification.

  **Why Self.xyz?**
  - **Zero-knowledge proofs**: Users prove identity without sharing raw documents
  - **Selective disclosure**: Users control exactly what information to share
  - **Privacy-first**: No raw identity documents are stored anywhere
  - **Compliance-ready**: Meets regulatory requirements while preserving privacy

  **What users need**: The free Self app (iOS/Android) to complete verification.
</Accordion>

  **For end users**: The Self app is free to download and use. Users don't pay anything for identity verification.

  - National ID cards (varies by country)

  ```typescript
    showVerificationOffer();
  }
  ```

  - **Optional upgrade**: Offer verification as an enhancement
  - **Required for new features**: Require verification for premium features
  - **Gradual enforcement**: Start optional, then require for sensitive operations

## Security and Privacy Questions

<AccordionGroup>
<Accordion title="How secure is the identity verification process?">
  **Security measures**:

  **Document Processing**:
  - All document analysis happens locally on the user's device
  - Raw documents never leave the user's phone
  - AI verification detects fraudulent or tampered documents

  **Data Transmission**:
  - Only zero-knowledge proofs are transmitted (not raw data)
  - All communications use HTTPS encryption
  - Cryptographic signatures prevent tampering

  **Storage**:
  - No raw identity documents stored anywhere
  - Only verified claims (not original data) recorded on blockchain
  - User maintains full control over their identity data

  **Privacy Protection**:
  - Users choose exactly what information to share
  - Selective disclosure prevents over-sharing
  - Zero-knowledge proofs mathematically verify claims without exposing data
</Accordion>

<Accordion title="What data is stored and where?">
  **What Trust Engine stores**:
  - Verification status (verified/pending/failed)
  - Cryptographic proofs (not readable data)
  - Requested disclosure fields (boolean flags)
  - Timestamps (creation, verification, expiration)

  **What Self.xyz stores**:
  - User account and wallet information (encrypted)
  - Verification proofs and signatures
  - No raw identity documents

  **What is NOT stored**:
  - Raw document scans or photos
  - Full identity information
  - Facial recognition data
  - Unencrypted personal information

  **Blockchain records**:
  - Cryptographic hashes only
  - No readable personal information
  - Immutable verification proofs
  - Privacy-preserving by design
</Accordion>

<Accordion title="Can verification data be deleted or updated?">
  **Data lifecycle**:

  **Verification sessions**:
  - Expire after 24 hours if not completed
  - Can be overwritten by new verification attempts
  - Failed sessions don't persist long-term

  **Verified identity data**:
  - Persists for the lifetime of the wallet relationship
  - Can be updated by completing new verification with different disclosure fields
  - Cannot be selectively deleted (maintain audit trail)

  **User control**:
  - Users can always create new verification sessions
  - Self.xyz users can manage their data through the Self app
  - Wallet relationships can be recreated if needed

  **Compliance**: 
  - Follows GDPR and data protection regulations
  - Users maintain rights over their identity data
  - Trust Engine provides data access and deletion mechanisms where legally required
</Accordion>

<AccordionGroup>
<Accordion title="How do I handle network failures during verification?">
  **Implement robust error handling**:

  ```typescript
  const robustVerification = async (walletAddress: string, disclosures: any) => {
    const maxRetries = 3;
    let attempt = 0;

    while (attempt < maxRetries) {
      try {
        const result = await sdk.verifyIdentity({ walletAddress, disclosures });
        return result;
      } catch (error) {
        attempt++;
        
        if (error.code === 'NETWORK_ERROR' && attempt < maxRetries) {
          console.log(`Attempt ${attempt} failed, retrying...`);
          await new Promise(resolve => setTimeout(resolve, 2000 * attempt));
          continue;
        }
        
        throw error; // Re-throw if not a network error or max retries reached
      }
    }
  };
  ```

  **Best practices**:
  - Implement exponential backoff for retries
  - Show user-friendly error messages
  - Allow manual retry options
  - Save progress where possible
</Accordion>

<Accordion title="Error: 'No verification session found'">
  **This error occurs when**:
  - User never started verification
  - Session expired (24 hours)
  - Wallet address doesn't match

  **Solution**:
  ```typescript
  const safeStatusCheck = async (walletAddress: string) => {
    try {
      return await sdk.checkIdentityStatus(walletAddress);
    } catch (error) {
      if (error.statusCode === 404) {
        // No session exists, user needs to start verification
        return { needsVerification: true };
      }
      throw error;
    }
  };
  ```

  **Prevention**: Always handle the case where no verification session exists.
</Accordion>
