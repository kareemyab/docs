## What This Endpoint Does

<Steps>
  <Step title="Transaction Deserialization">
    Decodes the Base64 transaction string and deserializes it into a Solana Transaction object
  </Step>
  <Step title="Server-Side Signing">
    Adds the server's fee payer signature to complete the transaction (partial signing)
  </Step>
  <Step title="Network Submission">
    Submits the fully-signed transaction to the Solana network using `sendRawTransaction`
  </Step>
  <Step title="Transaction Confirmation">
    Waits for and confirms the transaction on the blockchain with 'confirmed' commitment level
  </Step>
  <Step title="Result Formatting">
    Returns the transaction signature and explorer link for verification
  </Step>
</Steps>

---

## Parameters

<ResponseField name="transaction" type="string" required>
  Base64-encoded serialized Solana transaction
  
  **Constraints:** Must be a valid, pre-signed Solana transaction
  
  **Format:** Base64 string representation of serialized transaction bytes
  
  Example: `"AQAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAABAAEDArczbMia1tLmq2poQiNXU2xnkV4S4gK9bS6Kz..."`
</ResponseField>

## Understanding Transaction Flow

### Typical Workflow

<Tabs>
  <Tab title="Standard Wallet Flow">
    **Client Side:**
    1. Create transaction with instructions
    2. User signs transaction with their wallet
    3. Serialize and Base64 encode the signed transaction
    4. Send to `/submit-transaction` endpoint
    
    **Server Side:**
    1. Deserialize the transaction
    2. Add server's fee payer signature
    3. Submit to Solana network
    4. Confirm and return signature
  </Tab>
  
  <Tab title="Crossmint Wallet Flow">
    **Crossmint Integration:**
    1. Server creates transaction for user
    2. Crossmint API signs the transaction
    3. Server receives signed transaction
    4. Server uses `/submit-transaction` to finalize
    
    **Note:** This endpoint handles the final submission step regardless of signing method.
  </Tab>
</Tabs>

### Transaction States

<Info>
**Pre-Submission**: Transaction is signed but not yet on the blockchain

**Submitted**: Transaction sent to network but not yet confirmed

**Confirmed**: Transaction confirmed on blockchain with finality
</Info>

## Integration Examples

### With Register Endpoint

When registering files with standard wallets, the flow typically involves:

1. `/register` endpoint returns a prepared transaction (Base64)
2. Client signs the transaction with their wallet
3. Client calls `/submit-transaction` with the signed transaction

```typescript
// After receiving prepared transaction from /register
const signedTransaction = await wallet.signTransaction(preparedTransaction);
const serialized = signedTransaction.serialize();
const base64Transaction = Buffer.from(serialized).toString('base64');

const result = await fetch('/submit-transaction', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ transaction: base64Transaction })
});
```

### Error Handling Best Practices

```typescript
try {
  const result = await fetch('/submit-transaction', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ transaction: base64Transaction })
  });
  const data = await result.json();
  console.log('Transaction confirmed:', data.details.transactionSignature);
} catch (error) {
  if (error.message.includes('simulation failed')) {
    console.error('Transaction would fail - check account balances and permissions');
  } else if (error.message.includes('Missing')) {
    console.error('Invalid request format');
  } else {
    console.error('Network or server error:', error.message);
  }
}
```
